Keep View Logic Out of URLConfs

  • 事实上，我们从开发人员那里听说，看到在urlconf中定义的cbv是他们避免使用它们的部分原因。
  • Don’d repeat yourself
  • loose coupling:我们已经从urlconf中删除了模型和模板名称，因为视图应该是视图，urlcons应该是urlconf。我们应该能够从一个或多个urlconf调用我们的视图，我们的方法允许我们这样做。
  • uslconfs should do one thing and do it well:现在，我们的urlconf主要关注一件事：URL路由。我们没有跟踪视图和urlconf的视图逻辑，我们只是查看视图
  • our views benefit from being class-based:我们的视图，通过在视图模块中有一个正式的定义，可以继承其他类。这意味着添加身份验证、授权、新的内容格式或任何其他业务需求，我们的方式更容易处理。
  • infinite flexibility,无限的灵活性,通过在视图模块中有一个正式的定义，我们的视图可以实现自己的自定义逻辑。
  • ytiUse URL Namespaces
  • Try to Keep Business Logic Out of Views
  • Don’t Use locals() as Views Context

FBV

  • 使用基于函数的视图
    • 更少的视图函数
    • 没有重复
    • 视图应该处理表示逻辑。在可能的情况下，尽量在模型中保持业务逻辑，如果必须的话，则保持业务逻辑。
    • 让你的视图简单
    • 使用它们编写自定义403, 404和500错误处理程序
    • 复杂的嵌套-如果块是要避免的。
  • 通过 HttpRequest 对象

有时，我们希望在视图中重用代码，但不希望将其绑定到诸如中间件或上下文处理器之类的全局操作中可以跨项目使用的实用程序函数。
对于很多功能函数，我们从django.http.httRequest(或HttpRequest，简称HttpRequest)对象获取一个或多个属性，并收集数据或执行操作。

我们发现，通过将请求对象本身作为主参数，我们在更多方法上有更简单的参数。这意味着管理函数/方法参数的认知过载减少：只需传入HttpRequest对象！

SQLAlchemy Lazy属性

select (True ==‘select’)

  当首次访问属性时，使用单独的SELECT语句或简单的多对一引用的关系映射获取来加载元素。

immediate

  使用单独的SELECT语句，在加载父项时加载项目时加载子元素，或对简单的多对一引用进行关系映射。

joined (False == ‘joined’)

   应该使用“JOIN”或“LEFTJOIN”在与父连接相同的查询中积极地加载项目。查询根据内联接参数决定的。

subquery

   子查询-当加载父级时，应该加载一个项目，使用一个附加的SQL语句，为每个请求的集合发出一个连接到原始语句的子查询。

selectin

   在加载父级时，应使用“一个或多个附加SQL语句”加载项目，这些语句会向即时父对象发出联接，使用IN子句指定主键标识符。

noload (None == ‘noload’)

   不应在任何时间发生负载。这是为了支持“只写”属性，或者以特定于应用程序的方式填充的属性。

如果在通过浏览器访问视图函数的时候需要指定具体的参数个数类型,需要使用自定义转换器
自定义转换器的格式:

  1. 自定类,继承BaseConverter
  2. 重写init方法,接受三个参数
  3. 重写父类方法
  4. 将自定义的类,设置到app.url_map.converters[“名字”]

class MyConverterRegex(BaseConverter):

#自定义转化规则,需要将规则的url_map设置到父类
def __init__(self,url_map,regex):
    super(MyConverterRegex, self).__init__(url_map)
    self.regex = regex

 #将自定义的类,设置到app.url_map.converters["名字"]
app.url_map.converters["re"] = MyConverterRegex

准确性&精确性

   互联网并不存在真正准确的数据，不要过度追求数据的质量。

   收集数据有两种情况：准确的，精确的。

   精确性作为网站分析数据的战略，能够精确可预测，重现性好，能够根据分析得到更加自信的采取行动。

“如果你知道每一次设计时命中的位置，那么下一次开火的时候，你就能预测会发生什么。”

   大多数公司都痴迷于数据的准确性，对于准确性更重要的是平衡成本和收益，但是随着网络发展的加快，需要我们更块地作出决定，而精确的思维模式能够让决策者更快的行动起来。

数据质量的处理

   1. 收集简洁的数据

  总是使用第一方cookie，确保所有页面都正确加码；确保网站分析工具的权限设置正确。

   2.要有针对性

   网上所能收集到的数据量没有上限，尽可能收集现在和不久后将来需要的数据；

测试入门之A/B测试和MVT

A/B测试，用来测试网站上的某个网页两个或者多个版本优劣的方法。
   A/B测试的优点，性价比最高的一种，简单易行，快速实施；易于交流，不用但系多变量测试和回归等复杂技术。

   A/B测试的缺点：测试结果只有是或者不是，不能找到具体哪些改动发挥的作用。

MVT
MVT是指在一个页面当中同事测试多个不同元素的变化。
   根据浏览页面时，页面将动态地创建由不同元素构成，不同的图片，不同的行为引导方案…MVT就会开始收集数据；

   常用的MVT技术:全因子测试和部分因子测试。全因子测试能够带来非常充足的依据，代价是需要充足的时间保障和大量的被测试者来保证每一种组合获取足够的观测数据；部分因子测试以部分数据来获取很快的结果。此外还有就是使用部分因子先去除最差的组合，然后使用全因子来处理剩下的组合。

   MVT的优点：能够更好的帮助对访客偏好的理解，并且能够直接转化成更好的网站用户体验。
   MVT的缺点：他需要更多的组织团队的支持，而且他和A/B测试一样只是针对单个页面的测试。
